# pytest_examples

Simple examples to teach what unit tests are and how to run them with pytest.

## What Is A Unit Test?

A unit test checks one small piece of code (a "unit") and verifies that it behaves as expected.

In this repo, the units are two tiny functions in `src/example.py`.

## Quick Start (uv)

1. Install uv: https://docs.astral.sh/uv/getting-started/installation/
2. Sync development dependencies:

```bash
uv sync --dev
```

3. Run all tests:

```bash
uv run pytest
```

## Core Pytest Features In This Repo

### 1) Discovery

Pytest discovers tests by naming conventions.

- Files named `test_*.py`
- Functions named `test_*`
- Tests under the `tests/` folder

Try:

```bash
uv run pytest -q
```

### 2) Markers

Markers let you select groups of tests.

This repo uses `@pytest.mark.smoke` on one test.

Run only smoke tests:

```bash
uv run pytest -m smoke
```

### 3) Fixtures

Fixtures provide reusable setup for tests.

This repo has one fixture in `tests/conftest.py` that provides sample user data.

### 4) Hooks

Hooks let you customize pytest behavior.

This repo includes a tiny hook (`pytest_report_header`) in `tests/conftest.py`.
It adds a short header line when tests start.

## Automation In GitHub Actions

CI is configured in `.github/workflows/pytest.yml`.

### When It Runs

The workflow runs on:

- Every `push`
- Every `pull_request`

### Where It Runs

It runs on a fresh Linux runner: `ubuntu-latest`.
Every run starts clean, so dependencies are installed from scratch each time.

### What Each Step Does

On every run, GitHub Actions will:

1. Check out the code
2. Install Python
3. Install uv
4. Sync dependencies
5. Run `uv run pytest`

### What Success Or Failure Means

- If tests pass, the workflow is green.
- If tests fail, the workflow is red and the check appears as failed on the pull request.

## Why This Structure Works In Any Repo

- `src/` keeps application code separate from tests.
- `tests/` keeps tests discoverable and easy to scale.
- `__pycache__/` folders are auto-generated by Python to store bytecode cache files, which can speed up imports.
- These cache folders are not hand-written source code and can usually be ignored while learning project structure.
- `pyproject.toml` centralizes pytest settings and marker definitions.
- CI workflow guarantees tests run automatically before merge.
